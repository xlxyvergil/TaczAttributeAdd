## TACZ属性扩展到配件系统 - 整体流程

### 目标属性
1. **装填系统** (GunReloadData)
2. **热管理/过热系统** (GunHeatData) 
3. **近战攻击系统** (GunMeleeData)
4. **子弹物理特性-子弹数量** (GunData.ammoAmount)

### 原版数据模型分析

#### 1. 装填系统 (GunReloadData)
- **数据结构**: 包含type、feed、cooldown、infinite字段
- **feed/cooldown**: GunReloadTime结构，包含empty、base、minimum、maximum时间
- **特点**: 复杂的装填流程，支持空仓/战术换弹切换
- **使用方式**: 通过LivingEntityReload类管理装填状态和逻辑

#### 2. 热管理系统 (GunHeatData)
- **数据结构**: 包含heatMax、heatPerShot、coolingMultiplier、coolingDelay等
- **特性**: 支持热衰减曲线，影响射击精度和射速
- **使用方式**: 通过LivingEntityHeat类每tick更新热状态
- **数据流**: 影响射速(rpm)和精度(inaccuracy)

#### 3. 近战攻击系统 (GunMeleeData)
- **数据结构**: 包含distance、cooldown、defaultMeleeData
- **defaultMeleeData**: 支持自定义近战伤害和效果
- **使用方式**: 通过LivingEntityMelee类管理近战攻击

#### 4. 子弹数量 (GunData.ammoAmount)
- **数据结构**: 简单int类型，支持extendedMagAmmoAmount扩展
- **特性**: 弹匣容量，是基础属性

### 配件系统架构分析

#### 核心组件
1. **IAttachmentModifier<T, K>**: 配件修改器接口
2. **AttachmentPropertyManager**: 注册和管理所有modifier
3. **AttachmentCacheProperty**: 缓存计算结果的容器
4. **GunProperties**: 类型安全的属性键枚举

#### 工作流程
1. **初始化**: 通过initCache()获取枪械默认值
2. **收集**: 遍历所有配件，收集对应modifier的数据
3. **计算**: 通过eval()方法计算最终值
4. **缓存**: 结果存储在AttachmentCacheProperty中
5. **使用**: 通过getCache()方法获取计算后的值

### 扩展方案

#### 1. 创建新的Modifier类
为每个目标属性创建对应的IAttachmentModifier实现：
- **ReloadModifier**: 处理装填时间相关属性
- **HeatModifier**: 处理热管理相关属性  
- **MeleeModifier**: 处理近战攻击相关属性
- **AmmoCountModifier**: 处理子弹数量属性

#### 2. 注册到配件系统
在AttachmentPropertyManager.registerModifier()中添加：
```java
MODIFIERS.put(ReloadModifier.ID, new ReloadModifier());
MODIFIERS.put(HeatModifier.ID, new HeatModifier());
MODIFIERS.put(MeleeModifier.ID, new MeleeModifier());
MODIFIERS.put(AmmoCountModifier.ID, new AmmoCountModifier());
```

#### 3. 定义GunProperty枚举
在GunProperties类中添加：
```java
public static final GunProperty<GunReloadTime> RELOAD_TIME = GunProperty.of("reload", GunReloadTime.class);
public static final GunProperty<GunHeatData> HEAT_DATA = GunProperty.of("heat", GunHeatData.class);
public static final GunProperty<GunMeleeData> MELEE_DATA = GunProperty.of("melee", GunMeleeData.class);
public static final GunProperty<Integer> AMMO_COUNT = GunProperty.of("ammo_count", Integer.class);
```

#### 4. 数据访问策略
**关键要点**: 由于AttachmentPropertyEvent在配件计算前触发，必须提供完全计算后的最终值给TACZ系统使用，而不是中间值。

### 技术难点和注意事项

#### 1. 类型安全
- 使用GunProperty进行类型安全访问
- 避免原始的getCache(String)方法，防止ClassCastException

#### 2. 数据结构复杂性
- **ReloadTime**: 需要处理复杂的嵌套结构
- **HeatData**: 需要考虑热衰减曲线的影响
- **MeleeData**: 需要支持自定义伤害效果

#### 3. 事件系统时序
- 配件计算在事件之后进行
- 必须确保提供的是最终计算值，而非中间值

#### 4. 静态注册限制
- 需要通过mixin扩展AttachmentPropertyManager以支持动态注册
- 或者直接修改原版代码（如果允许）

### 实施步骤

1. **分析阶段**: 深入理解每个属性的数据结构和使用场景
2. **设计阶段**: 设计modifier的数据流和计算逻辑
3. **实现阶段**: 创建具体的modifier实现类
4. **集成阶段**: 注册到配件系统并测试
5. **优化阶段**: 优化性能和用户体验

### 优先级排序
1. **高优先级**: AmmoCountModifier (简单数值类型)
2. **中优先级**: MeleeModifier (中等复杂度)
3. **中优先级**: HeatModifier (热衰减逻辑)
4. **低优先级**: ReloadModifier (复杂嵌套结构)

### 总结
通过扩展TACZ的配件系统，可以让玩家通过配件来调整这些原版不支持修改的属性，大大增强游戏的可定制性。需要特别注意事件系统的时序和数据结构的复杂性，确保提供的是最终计算值。
